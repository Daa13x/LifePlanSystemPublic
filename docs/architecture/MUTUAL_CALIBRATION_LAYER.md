# Mutual Calibration Layer

## Core idea

LifePlanSystem should become a calibrated partnership between the user and the model.

The user steers the model so it understands them better.

The model helps the user compare current thinking against evidence, values, goals, constraints, and previous outcomes.

Neither side is assumed to be perfectly right.

Both sides update through evidence, correction, review, and repeated outcomes.

## Not a clone

The goal is not to make the assistant become the user.

The goal is to build a model that understands the user well enough to help them think, decide, recover, plan, and self-correct more accurately over time.

## The model learns

The system should learn, with confidence levels:

- what the user values;
- what the user avoids;
- what helps the user act;
- what advice formats work or fail;
- what emotional states change decisions;
- what contexts distort judgement;
- what recurring goals, fears, hopes, and constraints matter;
- what the user later says felt accurate or inaccurate.

## The user learns

The system should help the user notice patterns such as:

- repeated underestimation or overestimation;
- state-dependent urges;
- repeated avoidance loops;
- recurring decision traps;
- principles the user endorses but temporarily forgets;
- when previous beliefs should be updated by newer evidence.

## Understanding calibration

The system should track not only facts about the user, but also how confident it is in its understanding of different areas.

Suggested areas:

- reasoning style;
- career motivations;
- communication style;
- humour;
- health/accessibility model;
- execution/admin model;
- long-term priorities;
- risk tolerance;
- decision style;
- advice usefulness.

Low confidence should trigger questions or caution, not confident advice.

High confidence should still remain correctable.

## Advice calibration

The system should learn which kinds of advice actually help the user.

Track where practical:

- what advice was given;
- predicted usefulness;
- whether the user used it;
- whether it helped, failed, overwhelmed, annoyed, or missed the point;
- what should change next time.

Advice that sounds reasonable but repeatedly fails in practice should be revised or downgraded.

## Anti-sycophancy interaction

The user's history is evidence, not authority.

The model's interpretation is a hypothesis, not authority.

The model should not say: "You usually think X, therefore X is right."

It should say: "You often think X in this state. The evidence for X is mixed. Your deeper rule about evidence and risk may point toward Y."
